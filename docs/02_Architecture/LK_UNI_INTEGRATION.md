## 1. Решение

Сервис **Lk_uni** принимается как базовая Identity & Auth Platform для будущей системы регистрации, входа, восстановления доступа, ролей, сессий и аудита DevHub.

Lk_uni не встраивается напрямую в ядро DevHub на этапе Alpha. Он подключается как отдельный внешний сервис через REST API / SDK / Widget.

## 2. Причина

DevHub развивается в сторону командной и корпоративной платформы. Для этого необходимы:

- регистрация участников;
- вход;
- восстановление доступа;
- роли;
- права;
- сессии;
- аудит;
- журнал действий;
- проектные настройки авторизации.

Эти возможности уже заложены в Lk_uni как самостоятельной Identity & Auth Platform.

## 3. Архитектурная модель

```text
DevHub
  |
  | REST API / SDK / Widget
  v
Lk_uni
  |
  v
PostgreSQL
```

## 4. Роль Lk_uni в DevHub

Lk_uni отвечает за:

- регистрацию;
- вход;
- восстановление доступа;
- подтверждение пользователя;
- хранение способов идентификации;
- управление сессиями;
- аудит входов и критичных действий;
- проектные настройки авторизации.

DevHub отвечает за:

- проекты;
- документы;
- инженерную память;
- AI Hub;
- действия над проектами;
- качество;
- релизы;
- пользовательский интерфейс.

## 5. Принцип интеграции

Lk_uni является отдельным сервисом, а не частью ядра DevHub.

Это позволяет:

- использовать Lk_uni повторно в других проектах;
- не перегружать DevHub логикой авторизации;
- развивать авторизацию независимо;
- подключать разные каналы идентификации;
- масштабировать корпоративные сценарии.

## 6. Каналы идентификации

Базовые каналы:

- email;
- телефон;
- MAX;
- Telegram;
- VK ID;
- SberID.

Принцип: один пользователь может иметь несколько способов входа.

```text
User
  └── Identities
        ├── email
        ├── phone
        ├── max
        ├── telegram
        ├── vk
        └── sberid
```

## 7. DevHub Project Awareness

Каждый проект DevHub может иметь собственные настройки авторизации:

- включённые провайдеры;
- обязательные каналы подтверждения;
- правила регистрации;
- правила восстановления доступа;
- политики токенов;
- branding;
- allowed redirect URLs.

## 8. Минимальный сценарий для DevHub Beta

1. Пользователь открывает DevHub.
2. DevHub проверяет наличие активной сессии.
3. Если сессии нет — открывается экран входа.
4. Вход выполняется через Lk_uni.
5. DevHub получает access token.
6. DevHub вызывает `/api/auth/me`.
7. DevHub получает пользователя и список identities.
8. DevHub назначает локальный профиль и роль.
9. Все входы и критичные действия пишутся в журнал событий.

## 9. Роли DevHub

Первичный набор ролей:

- Администратор;
- Владелец продукта;
- Руководитель проекта;
- Архитектор;
- Разработчик;
- Аналитик;
- Специалист по качеству;
- Наблюдатель.

## 10. Что не делаем в Alpha

На этапе Alpha не внедряем полноценную авторизацию в Desktop-приложение.

В Alpha фиксируем архитектуру, требования и интерфейсы.

Реальная интеграция начинается после стабилизации Центра проектов.
