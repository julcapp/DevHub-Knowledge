## 1. Назначение

DevHub создаётся как инженерная платформа для управления полным жизненным циклом программных и инженерных проектов.

DevHub должен быть понятен не только разработчикам, но и руководителям проектов, аналитикам, специалистам по качеству и владельцам бизнеса.

## 2. Главная сущность

Главная сущность DevHub — **Проект**.

Пользователь работает с проектом, а не с Git, репозиторием или отдельными файлами.

Git, GitHub, AI, документация и релизы являются сервисами вокруг проекта.

## 3. Язык интерфейса

Пользовательский интерфейс DevHub должен использовать понятные русские термины.

Технические англоязычные термины допускаются только там, где они необходимы для разработчиков.

## 4. Инженерная память

DevHub должен сохранять инженерные знания:

- архитектурные решения;
- причины решений;
- требования;
- риски;
- историю поставок;
- рекомендации ИИ;
- результаты проверок.

Ни одно утверждённое решение не должно оставаться только в переписке.

## 5. AI

ИИ в DevHub помогает инженеру, но не заменяет его.

ИИ может:

- анализировать;
- объяснять;
- искать информацию;
- готовить документы;
- предлагать варианты.

ИИ не должен самостоятельно принимать критические инженерные решения без подтверждения человека.

## 6. Действия вместо пассивной информации

Любой элемент интерфейса должен помогать пользователю выполнить следующее полезное действие.

Информация без действия допускается только тогда, когда она нужна для принятия решения.

## 7. Прослеживаемость

Каждое требование, решение, архитектурное изменение и поставка должны быть связаны между собой.

Цепочка трассируемости:

```text
Требование → Решение → ADR → EDP/FP → Commit → Release
```

## 8. Документация как часть продукта

Документация DevHub является частью продукта, а не приложением к нему.

Документ считается полезным только если он помогает:

- разрабатывать DevHub;
- использовать DevHub;
- сопровождать DevHub;
- сохранять инженерные знания.

## 9. Принцип долгосрочной устойчивости

DevHub проектируется как продукт, который должен развиваться годами.

Быстрые решения допустимы только если они не создают критического технического долга.

## 10. Изменение Конституции

Изменение данного документа допускается только через архитектурное решение ADR и утверждение Product Owner.
