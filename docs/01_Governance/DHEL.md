# DevHub Engineering Lifecycle

**Документ:** DHEL.md  
**Код:** GOV-0001  
**Версия:** 1.0.0-alpha  
**Статус:** Approved  
**Ответственный:** CTO

## 1. Назначение

DHEL — стандарт жизненного цикла разработки DevHub.

Он определяет, как идея превращается в инженерную поставку, фиксируется в GitHub, проверяется и становится частью продукта.

## 2. Основной цикл

```text
Идея
  ↓
Анализ
  ↓
Требование
  ↓
Архитектурное решение
  ↓
Разработка
  ↓
Проверка
  ↓
Поставка
  ↓
GitHub
  ↓
Релиз
  ↓
Сопровождение
```

## 3. Типы поставок

| Тип | Назначение |
|---|---|
| FP | Foundation Pack: документационная и инженерная база |
| EDP | Engineering Delivery Package: код, документы, тесты, установка |
| ADR | Архитектурное решение |
| DHS | Стандарт DevHub |
| REL | Релиз продукта |

## 4. Definition of Ready

Поставка готова к разработке, если определены:

- цель;
- ожидаемый результат;
- затронутые репозитории;
- влияние на архитектуру;
- влияние на пользователя;
- критерии приёмки.

## 5. Definition of Done

Поставка считается завершённой, если:

- изменения внесены;
- документы обновлены;
- решения зафиксированы;
- GitHub синхронизирован;
- результат проверен;
- нет незакрытых обязательных решений.

## 6. Правило фиксации

Любая утверждённая договорённость должна попасть в GitHub.

Если решение не зафиксировано в GitHub, оно не считается реализованным.

## 7. Приоритет продукта

Документация не должна тормозить продукт.

Рабочий баланс после Foundation:

- 80% — код, интерфейс, тесты, функциональность;
- 20% — актуализация документации.
