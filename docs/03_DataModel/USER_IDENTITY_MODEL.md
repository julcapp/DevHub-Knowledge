## 1. Назначение

Документ фиксирует целевую модель пользователя и идентификации для DevHub с учётом интеграции Lk_uni.

## 2. Главный принцип

**Пользователь — единая сущность.**

Email, телефон, MAX, Telegram, VK ID, SberID и другие способы входа — это связанные identities одного пользователя, а не разные аккаунты.

## 3. Сущности

### 3.1. User

Единая сущность человека или аккаунта.

Поля:

| Поле | Назначение |
|---|---|
| id | Уникальный идентификатор |
| display_name | Отображаемое имя |
| first_name | Имя |
| last_name | Фамилия |
| status | Статус пользователя |
| profile | Расширенный профиль |
| consents | Согласия пользователя |
| created_at | Дата создания |
| updated_at | Дата обновления |
| deleted_at | Дата удаления / деактивации |

Статусы:

- draft;
- pending_verification;
- verified;
- active;
- blocked;
- deleted.

### 3.2. Identity

Способ идентификации пользователя.

Примеры:

- email;
- phone;
- max;
- telegram;
- vk;
- sberid.

Поля:

| Поле | Назначение |
|---|---|
| id | Уникальный идентификатор identity |
| user_id | Связанный пользователь |
| provider | Тип провайдера |
| provider_user_id | Идентификатор пользователя у провайдера |
| normalized_value | Нормализованное значение |
| verified | Подтверждён ли канал |
| verified_at | Дата подтверждения |
| metadata | Дополнительные данные |
| linked_at | Дата привязки |

### 3.3. Session

Активная пользовательская сессия.

Поля:

| Поле | Назначение |
|---|---|
| id | Уникальный идентификатор сессии |
| user_id | Пользователь |
| ip_address | IP-адрес |
| user_agent | Клиент |
| device_name | Устройство |
| status | Статус сессии |
| created_at | Дата создания |
| last_seen_at | Последняя активность |
| revoked_at | Дата отзыва |

### 3.4. ProjectMember

Участник проекта DevHub.

Поля:

| Поле | Назначение |
|---|---|
| id | Уникальный идентификатор |
| project_id | Проект DevHub |
| user_id | Пользователь |
| role | Роль в проекте |
| status | Статус участия |
| invited_at | Дата приглашения |
| joined_at | Дата подключения |

## 4. Роли DevHub

Первичный набор ролей:

| Роль | Назначение |
|---|---|
| Администратор | Управление системой, пользователями и настройками |
| Владелец продукта | Управление стратегией и приоритетами |
| Руководитель проекта | Управление проектом, статусами и участниками |
| Архитектор | Архитектурные решения, ADR, структура |
| Разработчик | Код, Git, поставки, технические задачи |
| Аналитик | Требования, процессы, документы |
| Специалист по качеству | Риски, аудит, соответствие стандартам |
| Наблюдатель | Просмотр без права изменения |

## 5. Связи

```text
User
  ├── Identity[]
  ├── Session[]
  └── ProjectMember[]
          └── Project
```

## 6. Аудит

Критичные действия должны фиксироваться в журнале событий:

- вход;
- выход;
- ошибка входа;
- восстановление доступа;
- смена роли;
- изменение прав;
- приглашение участника;
- блокировка пользователя;
- отзыв сессии.

## 7. Принцип безопасности

Пользователь не должен иметь доступ к проекту только на основании успешного входа.

Доступ определяется через связку:

```text
User → ProjectMember → Role → Permissions
```

## 8. Решение

Модель Lk_uni принимается как основа будущей модели пользователей, identities, сессий и ролей DevHub.
